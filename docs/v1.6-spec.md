# v1.6 Spec — RAG v2 Foundation

## 1. Purpose

Upgrade the current RAG system from simple top-k knowledge QA into a metadata-aware, source-cited, evidence/report/rule-aware retrieval foundation.

v1.6 improves retrieval and citation quality. It does not make RAG a detection source, does not allow RAG to override deterministic verdicts, and does not introduce autonomous investigation behavior.

Important framing:

- v1.6 improves retrieval and citation quality.
- v1.6 does not make RAG a detection source.
- v1.6 does not allow RAG to override deterministic verdicts.
- v1.6 does not implement AnswerGuardrails yet.
- v1.6 does not implement Smart Router.
- v1.6 does not implement Investigation Planner.

This milestone prepares a better retrieval and citation foundation for report-aware analyst support while preserving the project safety model established in earlier versions.

## 2. Problem

The current RAG implementation is mostly traditional vector-search QA. That works for broad security explanations, but it is less reliable for report-specific and rule-specific analyst questions.

Current limitations:

- Retrieval may not be precise for report questions.
- Short IDs such as `EV-003`, `F-001`, `CMD-001`, `XSS-001`, and `T1059` are not ideal for pure vector search.
- Answers do not consistently expose source citations.
- RAG currently works more like knowledge explanation than report/rule/evidence-aware analyst support.
- User-visible answers need better traceability.

These limitations matter because report follow-up questions often ask about exact evidence, findings, rule IDs, or MITRE technique IDs. A vector-only retrieval path may miss the right document or return a related but imprecise explanation.

## 3. Goals

- Add KB frontmatter metadata for existing `report_explainer` docs.
- Add metadata-aware retrieval planning.
- Add exact ID lookup for:
  - EV-ID
  - F-ID
  - rule ID
  - MITRE technique ID
- Add source citation support.
- Add `AnswerWithSources` schema.
- Add Report Explainer v2 using source-cited answers.
- Add Rule Explainer v2 using YAML rule metadata + KB source citations.
- Keep RAG explanation-only.
- Keep deterministic final verdict authority.

## 4. Non-Goals

- No AnswerGuardrails in v1.6.
- No full RAG QA evaluation dataset in v1.6.
- No Smart Router.
- No LLM-based routing.
- No Investigation Planner.
- No new `ai_safety` KB folder in v1.6.
- No Graph RAG.
- No knowledge graph.
- No web crawling.
- No external threat intelligence API.
- No LLM-generated detection rule activation.
- No production SIEM/SOAR integration.

## 5. Proposed KB Metadata Format

Existing report explainer Markdown files should gain YAML frontmatter. The body remains normal Markdown.

Scope:

- v1.6 should update the existing 11 `report_explainer` docs first.
- v1.6 should not create a large new `ai_safety` folder.
- Metadata should improve retrieval planning and citation display.
- Metadata should not become a source of detection truth.

Example:

```markdown
---
doc_id: report.risk_level_decision
doc_type: report_explainer
applies_to:
  - Security Triage Report
  - Risk Level
  - Decision
related_tools:
  - report_followup
  - rag_security_qa
attack_types:
  - Possible Account Compromise
  - Brute Force
severity:
  - LOW
  - MEDIUM
  - HIGH
rule_ids:
  - CMD-001
mitre_techniques:
  - T1059
keywords:
  - monitor
  - risk level
  - decision
---
```

Planned fields:

| Field | Required | Description |
|---|---:|---|
| `doc_id` | Yes | Stable unique document identifier. |
| `doc_type` | Yes | Document class such as `report_explainer`, `attack_technique`, or `rule_explainer`. |
| `applies_to` | No | Report sections, concepts, or workflows this document supports. |
| `related_tools` | No | Local tools or modes that may use this document. |
| `attack_types` | No | Relevant attack type names used by reports or rules. |
| `severity` | No | Relevant severities: `LOW`, `MEDIUM`, `HIGH`. |
| `rule_ids` | No | Detection rule IDs discussed by the document. |
| `mitre_techniques` | No | MITRE technique IDs such as `T1059`. |
| `keywords` | No | Search terms, aliases, bilingual terms, and report phrases useful for metadata filtering. |

## 6. Retrieval Architecture

Proposed flow:

```text
User Question
  -> RAGIntentClassifier
  -> Exact ID Lookup
  -> Metadata Filter
  -> Vector Retrieval
  -> Source Assembly
  -> AnswerWithSources
```

Clarifications:

- `RAGIntentClassifier` should be rule-based in v1.6.
- Exact ID lookup should run before vector search.
- Metadata filter should narrow candidate docs.
- Vector retrieval remains Chroma-based.
- Source citations should be source path + heading level at first.
- RAG v2 remains behind explicit RAG/report/rule explanation flows.
- RAG v2 is not a general-purpose router and does not choose which project tool should run.

## 8. Exact ID Lookup

Short operational IDs should not rely on embedding similarity alone.

Supported ID patterns:

- Evidence IDs: `EV-\d+`
- Finding IDs: `F-\d+`
- Incident IDs: `INC-...`
- Rule IDs:
  - `XSS-001`
  - `SQLI-001`
  - `PATH-001`
  - `CMD-001`
- MITRE technique IDs:
  - `T1059`
  - `T1190`
  - other `T####` style technique IDs

Clarifications:

- EV/F lookup should use current incident context when available.
- Rule ID lookup should use YAML `DetectionRule` metadata.
- MITRE ID lookup should be metadata-only in v1.6, not a full ATT&CK knowledge base.
- ID lookup should be deterministic string matching with normalized casing where appropriate.
- If an ID is present in the user question but cannot be found, the answer should say that the specific ID was not found in available local sources instead of inventing a source.

## 9. AnswerWithSources Schema

v1.6 should introduce a typed answer envelope for RAG-backed responses.

Planned model:

```python
class AnswerWithSources(BaseModel):
    answer: str
    sources: list[str]
    evidence_ids: list[str] = []
    finding_ids: list[str] = []
    rule_ids: list[str] = []
    confidence: Literal["LOW", "MEDIUM", "HIGH"]
    limitations: list[str] = []
```

Rules:

- `answer` must not be empty.
- `sources` should not be empty for source-cited answers.
- `confidence` is not a security verdict.
- `limitations` should be used when evidence is incomplete.

Expected behavior:

- `answer` contains the user-facing explanation.
- `sources` lists local KB, report, or rule metadata sources used.
- `evidence_ids`, `finding_ids`, and `rule_ids` expose exact IDs referenced by the answer.
- `confidence` describes answer support from available sources, not detection confidence.
- `limitations` should be used when sources are incomplete, ambiguous, or missing.

## 10. Citation Policy

Use simple citations first:

```text
knowledge/blue_team/report_explainer/risk_level_decision.md#Decision
```

Rules:

- v1.6 citations are source path + optional heading.
- Do not require paragraph-level citations yet.
- Do not claim a citation supports more than it actually says.
- If no source supports the answer, say information is insufficient.
- Cite repository-local documents and rule files.
- Prefer stable paths and document IDs over vague references.
- Do not cite sources that were retrieved but not actually used.
- Do not fabricate citation paths or document IDs.

Example display:

```text
Sources:
- knowledge/blue_team/report_explainer/risk_level_decision.md#Decision
- detections/blue_team/command_injection/command_injection_basic.yml
```

## 11. Report Explainer v2

Report Explainer v2 should improve answers for:

- `EV-003 是什麼意思？`
- `為什麼是 MONITOR？`
- `Risk Level 跟 Decision 差在哪？`
- `我接下來要查什麼？`

Use:

- `report_explainer` metadata
- exact EV/F lookup
- `AnswerWithSources`
- source citations

Expected capabilities:

- Explain why a risk level or decision appears in a report.
- Explain evidence IDs and finding IDs when present in the report context.
- Connect report concepts to relevant KB docs.
- Show citations for KB docs and report-derived evidence.
- State limitations when the report does not contain enough information.

Safety boundary:

- Report Explainer v2 may explain deterministic report output.
- It must not recalculate the final verdict.
- It must not change risk level, finding severity, or final recommendation.
- It must not claim that RAG discovered new evidence.

## 12. Rule Explainer v2

Rule Explainer v2 should explain why a YAML rule matched.

Use YAML rule metadata:

- rule ID
- attack type
- severity
- confidence
- patterns
- MITRE techniques
- references

Use KB citations if available.

Examples:

- `CMD-001 為什麼命中？`
- `XSS-001 是什麼？`

Expected capabilities:

- Look up a rule by rule ID.
- Explain rule title, attack type, severity, confidence, match mode, and MITRE mappings from YAML metadata.
- Link rule references to KB source citations when available.
- Explain what a rule is intended to detect in analyst-friendly language.
- State limitations when rule metadata or referenced KB sources are missing.

Important boundaries:

- Rule Explainer explains existing rules.
- It does not generate new rules.
- It does not enable or modify rules.
- It must not treat RAG output as detection evidence.
- It must not override deterministic detector results.

## 13. Safety Boundaries

RAG v2 must not:

- override Risk Level
- override Decision
- claim detection authority
- claim real blocking occurred
- invent evidence IDs
- invent rule IDs
- generate or activate new detection rules
- provide offensive exploit construction steps

RAG v2 may:

- explain existing evidence
- explain existing findings
- explain existing rules
- cite KB docs
- cite YAML rule metadata
- state limitations
- suggest safe analyst review steps
