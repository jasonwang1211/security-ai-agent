# v1.9 Spec - Architecture Cleanup and Orchestration Contracts

## 1. Purpose

Define v1.9 as an architecture cleanup and orchestration contract milestone.

v1.9 is meant to reduce ambiguity around existing helper paths, controller boundaries, router behavior, and future orchestration work. It does not introduce new runtime automation.

Core framing:

- deterministic detector and policy output remain the final security authority
- LLM/RAG output remains advisory and explanation-only
- RAG is not a detection source
- BLOCK, MONITOR, and ALLOW remain simulated decisions
- no real firewall, WAF, SIEM, SOAR, cloud, or endpoint action is performed

## 2. Problem

Earlier phases added typed controller infrastructure, tool registration, RAG helper types, source-cited explanation helpers, answer guardrails, Smart Router preview behavior, and deterministic analyst suggestions.

These pieces are intentionally conservative, but the repository now has several staged or preview-only paths. Before adding more intelligence or automation, v1.9 should document ownership, orchestration contracts, permission boundaries, and future placement decisions.

The main problem is not missing capability. The main problem is preventing unclear ownership from turning preview helpers into implicit runtime automation.

## 3. Goals

1. Document ControllerAgent as deterministic orchestration infrastructure.
2. Document Smart Router as advisory route classification unless a later phase explicitly wires a safe runtime path.
3. Define a tool permission model for read-only, draft, review-required, simulated, and forbidden actions.
4. Define a workflow plan model for future planning without auto-execution.
5. Create a module ownership map plan for cleanup work.
6. Clarify where future Graph RAG and Knowledge Capture would belong if approved later.
7. Preserve existing CLI behavior and RAGQA runtime.
8. Keep AI out of final attack, Risk Level, and Decision authority.

## 4. Non-Goals

v1.9 does not include:

- Graph RAG implementation
- Knowledge Capture implementation
- LLM-based tool selection
- runtime auto-execution
- Smart Router as the default CLI auto-route
- replacement of RAGQA
- AI-based attack determination
- AI override of Risk Level or Decision
- automatic detection rule modification
- real firewall, WAF, SIEM, SOAR, cloud, or endpoint enforcement
- offensive automation
- new runtime agent loops
- new parallel RAG, router, detector, or policy modules

- detection rules: `modules/detection_rules.py`, `modules/rule_loader.py`
- detector: `modules/detector.py`
- incident models: `modules/types.py`
- controller types: `modules/controller_types.py`
- tool registry: `modules/tool_registry.py`
- skill catalog and wrappers: `modules/skill_catalog.py`, `modules/skill_wrappers.py`
- controller agent: `modules/controller_agent.py`

- RAG intent and retrieval planning: `modules/rag_intent.py`, `modules/rag_retrieval_planner.py`

v1.9 should plan cleanup around these owners instead of creating replacement modules. Any future implementation should extend the existing owner only when a phase explicitly permits code changes.

## 6. ControllerAgent Role Decision

`ControllerAgent` is deterministic orchestration infrastructure.

Its role is to dispatch only through explicit route or registered tool names. It must not infer security verdicts, select tools through an LLM, execute unapproved actions, or bypass deterministic detector and policy output.

For v1.9, the ControllerAgent decision is documentation-only:

- keep ControllerAgent as a typed dispatcher
- keep ToolRegistry as the source of registered callable tools
- keep final security decisions outside ControllerAgent
- do not turn ControllerAgent into an autonomous agent loop

## 7. Smart Router Role Decision

Smart Router remains advisory route classification and preview behavior.

It may identify an input kind, route name, confidence, and reason. It must not execute the selected route automatically, become the default CLI path, choose tools with an LLM, change Risk Level or Decision, or replace existing numbered CLI modes.

For v1.9, Smart Router should be treated as:

- preview-only unless a later phase explicitly wires a guarded path
- rule-based, not LLM-based
- routing guidance, not detection or enforcement
- subordinate to existing controller and tool permission boundaries

## 8. Tool Permission Model

Future orchestration should classify tools by permission level before any wiring work:

- `READ_ONLY`: deterministic retrieval, explanation, validation, or suggestion helpers
- `WRITE_DRAFT`: helpers that create draft artifacts without committing them to runtime state
- `WRITE_REVIEW_REQUIRED`: write or commit helpers that require explicit human review before use
- `SIMULATED_ACTION`: simulated decision or demo actions that do not perform real enforcement
- `FORBIDDEN`: real enforcement, autonomous deployment, offensive automation, or verdict override paths

Examples:

- `READ_ONLY`: `vector_rag_search`, `graph_lookup`, `report_explainer`, `rule_explainer`, `answer_guardrails_check`, `analyst_suggestions`
- `WRITE_DRAFT`: `knowledge_capture_draft`, `rule_draft_suggestion`
- `WRITE_REVIEW_REQUIRED`: `knowledge_ingest`, `graph_commit`
- `SIMULATED_ACTION`: `simulated_block`, `simulated_monitor`, `simulated_allow`
- `FORBIDDEN`: `real_firewall_block`, `real_waf_update`, `real_account_disable`, `auto_rule_activation`, `autonomous_rule_deployment`

Unknown tools should default to `FORBIDDEN` until explicitly registered and documented. LLM output must not upgrade a tool permission level.

## 9. Workflow Plan Model

Future workflow planning may describe steps, dependencies, expected inputs, and required confirmations.

Workflow execution modes should align with the tool permission model:

- `PREVIEW_ONLY`: describe a proposed route, plan, or action without executing it
- `READ_ONLY`: run deterministic read-only helpers only
- `HUMAN_APPROVAL_REQUIRED`: pause before any write, commit, external call, or review-required action
- `SIMULATED_ONLY`: allow simulated BLOCK, MONITOR, or ALLOW actions only
- `FORBIDDEN`: reject workflows that require real enforcement, autonomous deployment, or verdict override

v1.9 only defines the contract. It does not auto-execute workflow plans.

The model must remain plan-only by default:

- no runtime auto-execution
- no hidden tool calls
- no automatic remediation
- no automatic detection rule edits
- no final verdict changes

A workflow plan may recommend analyst next steps, but execution must remain explicit and bounded by the tool permission model.

## 10. Module Ownership Map Plan

v1.9 should produce or prepare a concise ownership map before any cleanup implementation.

The map should answer:

- which module owns each current capability
- which helpers are preview-only or staged
- which paths are user-facing
- which paths are test-only
- which future work would extend an existing owner rather than create a parallel module

The map should not move code by itself. Refactors should be split into later phases with focused tests.

## 11. Testing Strategy Plan

v1.9 testing should focus on contract preservation when later code changes are approved.

Planned coverage areas:

- ControllerAgent dispatch remains explicit and deterministic
- Smart Router preview does not execute tools
- tool permissions prevent forbidden or unknown actions
- workflow plans do not auto-execute
- RAGQA remains the active general knowledge QA runtime
- AI does not alter Risk Level, Decision, evidence, finding severity, or detector output
- unit tests do not initialize Ollama, Chroma, embeddings, Torch, ChatOllama, or local LLM clients

For this Phase 9A-0 spec task, validation is documentation and quality-gate focused.

## 12. Future Graph RAG Placement

Graph RAG is deferred beyond v1.9.

If approved later, Graph RAG should be placed as retrieval infrastructure under existing RAG ownership boundaries. It should not become a detector, policy engine, enforcement system, or replacement for RAGQA without a separate explicit phase.

Future Graph RAG must preserve:

- deterministic detector and policy final authority
- RAG as explanation and retrieval support only
- no attack decision by AI
- no Risk Level or Decision override
- no unit-test startup of heavyweight local LLM or embedding infrastructure

## 13. Future Knowledge Capture Placement

Knowledge Capture is deferred beyond v1.9.

If approved later, Knowledge Capture should be treated as a controlled documentation or knowledge-ingestion workflow, not automatic rule generation. It must not silently modify detection rules, promote analyst notes into policy, or update runtime knowledge without explicit review.

Future Knowledge Capture must preserve:

- analyst confirmation before any write or import
- clear provenance for captured material
- no automatic detection rule edits
- no runtime verdict changes
- no real enforcement integration
