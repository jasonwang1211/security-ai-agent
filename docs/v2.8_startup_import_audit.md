# v2.8-A Startup Import Audit and Lazy Loading Plan

## Scope

This document audits startup/import performance after v2.7.0 and proposes a focused lazy-loading plan for heavy RAG, embedding, vector, and LLM dependencies.

This is a read-only architecture audit. It does not change runtime behavior.

## Starting State

- Branch observed during audit: `main`
- HEAD observed during audit: `fdd08f1`
- Initial `git status --short --untracked-files=all`: clean
- Observed local disk/load evidence from user: `venv` is about `1.32 GB`, `.mypy_cache` is about `0.14 GB`, and the largest files are dependency binaries such as `torch_cpu.dll`, NumPy/SciPy OpenBLAS, PyArrow DLLs, ONNX Runtime, and gRPC binaries.

Working hypothesis: startup feels slower because heavy ML/vector/RAG packages are imported or initialized too early, not because project source/docs/git are large.

## Startup Path Findings

### CLI startup: `python app.py`

Current path:

1. `app.py` imports `RAGQA` at module import time.
2. Importing `modules.rag_qa` imports LangChain ChatOllama, HuggingFaceEmbeddings, Chroma, ChatPromptTemplate, and `RAGQueryPlanner`.
3. `main()` constructs `RAGQA()` immediately.
4. `RAGQA.__init__()` loads controlled metadata, builds prompt templates, initializes HuggingFace embeddings, initializes ChatOllama/query planner, and opens Chroma if available.
5. Only after that does the menu/direct-input loop begin.

Finding: CLI startup currently front-loads RAG/embedding/vector work even when the user only wants deterministic payload analysis.

### Streamlit startup: `python -m streamlit run ui/streamlit_app.py`

Current path:

1. `ui/streamlit_app.py` imports `RAGQA` at module import time.
2. `main()` calls `get_runtime()` during initial render.
3. `get_runtime()` calls `build_agent()` when session runtime is missing.
4. `build_agent()` constructs `RAGQA()` before any user action.
5. The console then renders controls and panels.

Finding: Streamlit currently pays RAG import and initialization cost on first page render, not only when Knowledge Q&A or full AI/RAG behavior is used.

### Fast deterministic analysis path

Current path:

1. `run_direct_input()` may choose `run_fast_payload_analysis()` when the selected mode is fast deterministic.
2. `_run_fast_payload_analysis_with_timing()` still calls `get_runtime()` to obtain the shared `SecurityAgent`.
3. `get_runtime()` builds a full agent with `RAGQA()` if no runtime exists yet.

Finding: fast deterministic mode avoids the full orchestrator path, but it does not avoid RAGQA import/construction on first use because the shared agent is built eagerly.

### Full AI-assisted / RAG path

Current path:

1. Full AI-assisted mode goes through the orchestrator and `SecurityAgent`.
2. RAGQA is already initialized by startup/runtime construction.
3. `LLMAssist` imports LangChain prompt support at module load, but ChatOllama construction is lazy inside `_initialize_llm()`.

Finding: full mode is expected to pay LLM/RAG costs, but the current design pays much of that cost before full mode is selected.

### Knowledge Q&A path

Current path:

1. CLI Mode 3 calls `run_knowledge_qa()` -> `agent.handle_knowledge_query()` -> `build_rag_answer()`.
2. Streamlit Knowledge Q&A forces the KnowledgeQASkill path through the orchestrator.
3. `build_rag_answer()` uses the already-constructed `rag_qa` object.

Finding: Knowledge Q&A is the correct place to initialize RAG, but RAG is currently initialized earlier than needed.

## Lazy-Loading Candidates

Highest-value candidates:

1. `RAGQA` import in `app.py` and `ui/streamlit_app.py`.
   - Replace eager direct imports with a small provider/factory or function-local import.
   - Keep type checking clean with `TYPE_CHECKING` if needed.

2. `RAGQA.__init__()` component initialization.
   - Split cheap state setup from heavy component initialization.
   - Defer HuggingFaceEmbeddings, ChatOllama/RAGQueryPlanner, and Chroma construction until `is_ready()`, `is_security()`, `retrieve_context()`, or `answer_question()` actually needs them.
   - Prefer an idempotent `_ensure_initialized()` guard.

3. `modules/rag_qa.py` top-level heavy imports.
   - Move `HuggingFaceEmbeddings`, `Chroma`, and `ChatOllama` imports inside the initialization method that needs them.
   - Keep Pydantic/project RAG helper imports at module level if they remain lightweight.

4. `modules/rag_query_planner.py` top-level ChatOllama import.
   - Move ChatOllama import inside `_get_llm()`.
   - Consider moving `ChatPromptTemplate` import into constructor only if import-weight tests show it matters.

5. `LLMAssist` prompt dependency.
   - Lower priority than RAGQA.
   - ChatOllama is already lazy; only consider delaying `ChatPromptTemplate` import/prompt construction if startup measurements still show noticeable LangChain cost after RAGQA changes.

6. `modules/ingest.py`.
   - Keep ingest heavy imports isolated from app/UI startup.
   - Avoid importing `modules.ingest` from normal runtime paths.

## Safe Change Recommendations

- Move heavy imports inside functions where possible, especially in `modules/rag_qa.py` and `modules/rag_query_planner.py`.
- Keep UI helper modules pure and lightweight. `modules/ui/*` currently avoid direct Chroma/Ollama/LangChain/Torch imports and should stay that way.
- Ensure Fast deterministic mode does not import or initialize RAG/LLM dependencies on first interaction.
- Ensure Knowledge Q&A initializes RAG only when used.
- Preserve `RAGQA.is_ready()` semantics, but allow it to initialize lazily or expose a cheap readiness state plus explicit initialization path.
- Consider a `LazyRAGQA` adapter or provider object that satisfies the `SecurityAgent` interface: `is_ready()`, `is_security()`, `answer_question()`, `retrieve_context()`, `generate_answer()`.
- Add import-weight regression tests that import fast-path modules in a clean subprocess and assert forbidden heavy modules are absent from `sys.modules`.
- Add fast startup smoke tests for importing `app`, importing `ui.streamlit_app`, and running fast deterministic analysis with stubs/mocks that prove `chromadb`, `sentence_transformers`, `torch`, and `langchain_community.vectorstores` are not imported.

## Risky Files To Avoid Casual Refactors

These files protect safety or provenance boundaries and should not be casually changed during import cleanup:

- `modules/detector.py`
- `modules/triage_policy.py`
- deterministic decision logic and responder/report decision wording
- `modules/rag_qa.py` answer safety framing sections, except narrowly for import placement and initialization timing
- `modules/controller/tool_policy.py`
- `modules/graph/*` provenance helpers
- `modules/controller/case_capture.py` and related case draft approval gate behavior
- approved similar-case retrieval and seed loading behavior

For `modules/rag_qa.py`, refactor only the import/initialization boundary first. Do not rewrite answer generation, safety framing, CVE/CVSS normalization, or controlled retrieval semantics in the same batch.

## Disk Hygiene Notes

Safe cleanup candidates when the user explicitly wants disk cleanup:

- `.mypy_cache`
- `.pytest_cache`
- `.ruff_cache`
- `.pytest_tmp`
- `__pycache__`

Cautious cleanup candidates:

- `venv`: large and rebuildable, but deleting it requires reinstalling dependencies; do only with explicit approval and clear recreate commands.
- `chroma_db`: may contain local vector index state; do not delete during release validation unless the user explicitly wants to rebuild the knowledge index.
- model caches: may be outside the repo and expensive to redownload; inventory first.
- screenshots / demo evidence: may be needed for portfolio/release evidence; archive or prune only after review.

Do not use cleanup as the primary startup-performance fix. Cache cleanup may free disk space, but lazy imports are the safer way to reduce perceived load cost.

## Proposed v2.8 Implementation Plan

1. v2.8-A Startup Import Audit
   - Complete this document.
   - Use it as the scope boundary for follow-up implementation.

2. v2.8-B Cache Cleanup Checklist
   - Document safe vs cautious cleanup commands.
   - Do not delete caches automatically.

3. v2.8-C Lazy RAG Import Refactor
   - Move RAGQA heavy imports and initialization behind a lazy guard.
   - Keep public behavior identical for Knowledge Q&A and full AI-assisted paths.
   - Avoid changing RAG safety framing or answer semantics.

4. v2.8-D Fast Deterministic Startup Smoke Tests
   - Add subprocess import tests for `app`, `ui.streamlit_app`, UI helpers, and fast deterministic analysis.
   - Assert heavy modules are not imported on fast startup.
   - Keep full RAG tests covering lazy initialization when Knowledge Q&A is used.

5. v2.8-E UI Helper Deduplication
   - Keep this separate from lazy-loading.
   - Only deduplicate pure UI helper code after startup behavior is protected by tests.

6. v2.8-F Release Gate
   - Run full validation after lazy-loading and cleanup docs are complete.
   - Repeat safety boundary checks: deterministic detector authority, deterministic Risk Level / Decision, simulated decisions, advisory-only RAG/LLM/AI Analyst/Evidence Gap, no real enforcement, no exploit/PoC/traffic generation, human review required.

## Recommended First Refactor Slice

For v2.8-C, the safest first slice is:

1. Introduce a lazy RAG provider or make `RAGQA` internally lazy without changing its external methods.
2. Move `HuggingFaceEmbeddings`, `Chroma`, and `ChatOllama` imports from `modules/rag_qa.py` top level into the guarded initialization method.
3. Move `ChatOllama` import in `modules/rag_query_planner.py` into `_get_llm()`.
4. Update `app.py` and `ui/streamlit_app.py` so they do not import or construct real `RAGQA` before RAG is needed.
5. Add regression tests before broad cleanup.

Expected outcome: fast CLI/UI startup and fast deterministic analysis should avoid loading Chroma, sentence-transformers, Torch, and related transitive binary stacks unless the user chooses Knowledge Q&A/full RAG behavior.
