# v1.7 Spec — Answer Safety, Evaluation, and Smart Router

## 1. Purpose

Define v1.7 as the reliability and routing milestone after the v1.6 RAG v2 Foundation.

v1.7 focuses on:

- Answer Safety
- evaluation datasets
- Smart Router
- RAG v2 runtime integration decision
- follow-up module ownership decision

Important framing:

- v1.7 should not start by adding more parallel RAG scaffolding.
- v1.7 should prefer evaluation, safety checks, ownership decisions, and controlled wiring.
- v1.7 keeps deterministic detector and policy output as the final security authority.
- v1.7 keeps LLM/RAG advisory and explanation-only.
- v1.7 keeps BLOCK, MONITOR, and ALLOW simulated.

## 2. Problem

v1.6 added RAG v2 helper infrastructure, including staged support for metadata-aware retrieval, exact ID lookup, source-cited answers, report explanations, and rule explanations.

The existing RAGQA runtime is still active and unchanged. RAG v2 helpers are staged and are not yet the primary user-facing runtime. This means the repository now has multiple RAG-related modules and paths with different maturity levels.

Before adding more features, v1.7 must define answer safety and evaluation boundaries. Smart Router should not be added before safety and evaluation constraints are specified, because routing expands the number of paths that can produce user-facing answers.

The follow-up modules also need an ownership decision:

- `followup_handler.py` remains point-based conversational follow-up in the CLI flow.
- `report_followup.py` remains report-aware EV/F-ID/incident follow-up.

v1.7 should decide whether these stay separate, unify behind one route, or move behind ControllerAgent/ToolSpec boundaries. The spec phase should document the choice space, not refactor the modules.

## 3. Goals

1. Add Answer Safety foundation.
2. Add evaluation dataset foundation.
3. Define RAG v2 runtime integration strategy.
4. Define follow-up module ownership strategy.
5. Add Smart Router only after safety/evaluation constraints are specified.
6. Keep deterministic final verdict authority.
7. Keep LLM/RAG advisory and explanation-only.
8. Keep BLOCK, MONITOR, and ALLOW simulated.

## 4. Non-Goals

v1.7 does not include:

- No autonomous investigation agent.
- No Investigation Planner implementation yet.
- No Advanced AnswerGuardrails yet.
- No real firewall, WAF, SIEM, or SOAR integration.
- No LLM-based routing.
- No LLM final verdict.
- No automatic rule generation or activation.
- No Graph RAG.
- No external threat intelligence API.
- No web dashboard.
- No major RAG module consolidation without tests and explicit decision.
- No new parallel RAG modules without clear ownership.

## 5. Proposed v1.7 Phase Plan

### Phase 7A-1 — Evaluation Dataset Spec and Minimal Cases

- Add `eval_cases/` only.
- Add small JSONL datasets.
- No runtime changes.
- No LLM calls.

### Phase 7A-2 — AnswerGuardrails Foundation

- Add deterministic answer safety checks.
- Check `AnswerWithSources`, source presence, unsafe claims, fake enforcement claims, and final verdict override claims.
- No LLM classifier.

### Phase 7A-3 — RAG v2 Runtime Integration Decision

- Decide whether to keep RAGQA as active runtime and RAG v2 staged.
- Decide whether to wire report/rule explainer helpers into `report_followup`.
- Decide whether to create a protected RAG v2 answer path.
- Decide whether to consolidate modules later after evaluation exists.
- This phase may be docs/design first.

### Phase 7A-4 — Follow-up Module Ownership Decision

- Decide whether `followup_handler.py` stays point-based conversational follow-up.
- Decide whether `report_followup.py` stays report-aware EV/F-ID follow-up.
- Decide whether both unify behind one ToolSpec/report-aware follow-up path.

### Phase 7A-5 — Smart Router Spec

- Define input categories and routing boundaries.
- No LLM routing.
- Router selects tools only.
- Router does not change Risk Level or Decision.

### Phase 7A-6 — Smart Router Minimal Implementation

- Rule-based only.
- Use ControllerAgent / ToolRegistry where appropriate.
- Keep existing CLI modes available.

### Phase 7A-7 — Documentation and Release Gate

- Sync docs.
- Run the full quality gate.
- Prepare the `v1.7.0` tag.

## 6. Answer Safety Foundation

Planned model names:

- `AnswerSafetyFinding`
- `AnswerSafetyReport`
- `AnswerSafetySeverity`
- `AnswerSafetyCheck`

AnswerGuardrails should check:

- `AnswerWithSources` has sources when claiming facts.
- Answer does not claim real firewall, WAF, SIEM, or SOAR enforcement.
- Answer does not say RAG is the detection source.
- Answer does not say LLM changed Risk Level or Decision.
- Answer does not turn MONITOR into confirmed compromise.
- Answer does not invent EV-ID, F-ID, or rule ID.
- Answer includes limitations when evidence is incomplete.
- Answer keeps RAG explanation-only.

v1.7 checks should be deterministic and testable. Do not use LLM-based safety classification in v1.7.

## 7. Evaluation Dataset Foundation

Proposed directory:

```text
eval_cases/
├── answer_safety_cases.jsonl
├── report_qa_cases.jsonl
├── router_cases.jsonl
└── payload_detection_cases.jsonl
```

Suggested fields for `answer_safety_cases.jsonl`:

- `id`
- `answer`
- `sources`
- `expected_findings`
- `forbidden_claims`
- `notes`

Suggested fields for `report_qa_cases.jsonl`:

- `id`
- `question`
- `expected_contains`
- `forbidden_contains`
- `expected_sources`
- `notes`

Suggested fields for `router_cases.jsonl`:

- `id`
- `user_input`
- `expected_input_kind`
- `expected_route`
- `notes`

Suggested fields for `payload_detection_cases.jsonl`:

- `id`
- `payload`
- `expected_status`
- `expected_attack_types`
- `expected_decision`
- `notes`

Keep datasets small in v1.7. This is regression testing, not a statistical benchmark yet.

## 8. Smart Router Scope

Smart Router should classify:

- `payload_or_event`
- `raw_log_line`
- `log_file_path`
- `security_knowledge_question`
- `report_followup`
- `incident_export`
- `unknown`

Rules:

- Smart Router is rule-based in v1.7.
- No LLM router.
- Router selects a tool or asks for clarification.
- Router must not change final verdict.
- Router must not call unregistered tools.
- Router should use ControllerAgent / ToolRegistry when wired.
- Existing CLI modes must remain available.

## 9. RAG v2 Runtime Integration Decision

Documented options:

### Option A — Keep RAGQA Active

Keep RAGQA as the active runtime and keep RAG v2 helpers staged.

### Option B — Wire Report/Rule Helpers Narrowly

Wire Report Explainer v2 / Rule Explainer v2 into `report_followup` only.

### Option C — Protected RAG v2 Answer Path

Create a protected RAG v2 answer path behind AnswerGuardrails.

### Option D — Defer Runtime Wiring

Defer runtime wiring until the evaluation dataset is mature.

Recommended default:

- Do not replace RAGQA immediately.
- Prefer evaluation plus AnswerGuardrails first.
- Then wire a narrow report/rule explanation path.

## 10. Follow-up Module Ownership Decision

Current split:

- `followup_handler.py`: point-based conversational follow-up in CLI flow.
- `report_followup.py`: report-aware EV/F-ID/incident follow-up.

v1.7 should decide whether to:

- keep both separate
- unify behind ToolSpec
- route both through ControllerAgent
- deprecate one after tests

Do not refactor this in the spec phase.

## 11. Safety Boundaries

Preserve:

- deterministic detector and policy own final verdict
- LLM/RAG are advisory/explanation-only
- RAG is not a detection source
- BLOCK, MONITOR, and ALLOW are simulated
- no real firewall, WAF, SIEM, or SOAR actions
- no offensive automation
- no automatic detection rule activation
- no LLM-based routing in v1.7

## 12. Test Strategy

v1.7 tests should cover:

- JSONL eval case loading
- AnswerGuardrails deterministic checks
- unsafe claim detection
- missing citation detection
- invented ID detection
- router case classification
- unknown input clarification
- ControllerAgent route compatibility
- no LLM / Chroma / Ollama startup in unit tests
- existing 366 tests still pass
