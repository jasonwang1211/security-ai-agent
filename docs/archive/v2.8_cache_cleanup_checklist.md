# v2.8-B Cache Cleanup Checklist

## Purpose

This checklist is a human-reviewed disk cleanup and repo-hygiene guide. It is not an automated cleanup script. Do not copy/paste commands blindly during a demo window; review the current repository state, decide what is safe to remove, and validate afterward.

Context from [v2.8_startup_import_audit.md](v2.8_startup_import_audit.md): project source, docs, and git data are small. Local disk usage is mostly from `venv` and tooling/model dependencies. Startup slowness is more likely caused by heavy RAG/embedding imports being loaded too early than by source code size.

## Safe Cleanup Candidates

| Path / pattern | What it is | Why safe to delete | Regenerated? | Suggested manual PowerShell command | Validation after cleanup |
|---|---|---|---|---|---|
| `.mypy_cache` | Type-check cache created by mypy. | It contains derived analysis data only. | Yes, next `python -m mypy ...` run rebuilds it. | `Remove-Item .mypy_cache -Recurse -Force -ErrorAction SilentlyContinue` | Run `python -m mypy app.py modules tests` when type-check validation is needed. |
| `.pytest_cache` | Pytest cache for node IDs, last-failed state, and run metadata. | It does not contain source truth. | Yes, next pytest run recreates it. | `Remove-Item .pytest_cache -Recurse -Force -ErrorAction SilentlyContinue` | Run a focused pytest smoke or the intended test command. |
| `.ruff_cache` | Ruff lint cache. | It only speeds up linting. | Yes, next `python -m ruff check ...` recreates it. | `Remove-Item .ruff_cache -Recurse -Force -ErrorAction SilentlyContinue` | Run `python -m ruff check .` or focused ruff command. |
| `.pytest_tmp` | Temporary pytest output directory used by local test runs. | It should contain disposable test artifacts. | Yes, test runs can recreate temp output as needed. | `Remove-Item .pytest_tmp -Recurse -Force -ErrorAction SilentlyContinue` | Run focused tests that use temp paths. |
| recursive `__pycache__` | Python bytecode caches under package/test directories. | Bytecode is derived from `.py` files. | Yes, Python recreates it automatically on import. | `Get-ChildItem -Recurse -Directory -Force -Filter "__pycache__" | Remove-Item -Recurse -Force -ErrorAction SilentlyContinue` | Start the app or run focused tests; imports recreate needed caches. |

## Safe Cleanup PowerShell Commands

Manual execution only. Review `git status` first and run from the repository root.

```powershell
Remove-Item .mypy_cache -Recurse -Force -ErrorAction SilentlyContinue
Remove-Item .pytest_cache -Recurse -Force -ErrorAction SilentlyContinue
Remove-Item .ruff_cache -Recurse -Force -ErrorAction SilentlyContinue
Remove-Item .pytest_tmp -Recurse -Force -ErrorAction SilentlyContinue
Get-ChildItem -Recurse -Directory -Force -Filter "__pycache__" | Remove-Item -Recurse -Force -ErrorAction SilentlyContinue
```

Recommended validation after safe cache cleanup:

```powershell
git status --short --untracked-files=all
python -m pytest tests/test_ui_demo_scenarios.py
python -m ruff check .
```

Use the focused smoke first when the goal is only to confirm the workspace still runs. Run the full release gate only when preparing a release.

## Cautious Cleanup Candidates

| Path / area | Why it may be large | Why not delete casually | How to rebuild or verify | Recommended action |
|---|---|---|---|---|
| `venv` | Contains installed Python dependencies, including ML/vector binary stacks such as Torch, NumPy/SciPy, Chroma, ONNX Runtime, and gRPC transitive binaries. | Deleting it breaks local commands until dependencies are reinstalled; rebuild may take time and network/cache availability. | Recreate with `python -m venv venv`, activate, install `requirements.txt` and `requirements-dev.txt`, then run a focused smoke. | Delete only after backup/time planning; do not delete right before a demo. |
| `chroma_db` | Local vector index for RAG knowledge retrieval. | Deleting it makes RAG unavailable until rebuilt. | Run `python ingest_knowledge.py`, then verify with a Knowledge Q&A smoke question. | Keep unless explicitly planning a RAG index rebuild. |
| Model caches | Embedding/LLM/runtime packages may cache model files outside or inside local environment paths. | They may be expensive to redownload and may be shared across projects. | Inventory location first; verify the app can reload models after cleanup. | Archive or delete only with explicit plan. |
| `docs/screenshots` | Demo screenshots and portfolio evidence can be image-heavy. | They are release/demo evidence and may be referenced by docs or slides. | Verify docs links and demo materials before removing. | Keep before demos; archive first if pruning. |
| Release/demo evidence docs | Reports, smoke checklists, release gates, and notes accumulate over versions. | They document validation history and safety boundaries. | Confirm the current release no longer references them. | Keep in repo unless intentionally archiving older release evidence. |
| Local backups | Backup directories can contain full copies or generated artifacts. | They may be the only copy of manual work not committed elsewhere. | Inspect content and compare to git history before removal. | Archive first; delete only after backup. |
| Exported reports / demo outputs if present | Generated reports or sample outputs may accumulate. | They may be used for demos, grading, or regression comparison. | Confirm references in docs and tests; regenerate if supported. | Archive first or delete only after review. |

## venv Rebuild Section

`venv` is the biggest repo-local disk consumer, but deleting it requires reinstalling dependencies. Do not delete `venv` right before a demo unless there is enough time to reinstall and retest.

Example rebuild sequence:

```powershell
Remove-Item venv -Recurse -Force
python -m venv venv
.\venv\Scripts\Activate.ps1
pip install -r requirements.txt
pip install -r requirements-dev.txt
python ingest_knowledge.py
python -m pytest tests/test_ui_demo_scenarios.py
```

Notes:

- `python ingest_knowledge.py` is needed if the RAG index must be rebuilt after environment or index cleanup.
- Run broader validation only when preparing a release gate.
- If package installation depends on network access, confirm network/cache availability first.

## Chroma / RAG Index Section

Do not delete `chroma_db` unless you are ready to rebuild the RAG index.

Rebuild command:

```powershell
python ingest_knowledge.py
```

After rebuild, verify with a Knowledge Q&A smoke question. Good examples include a defensive HTTP/2 DoS, Resource Exhaustion, CVE, or Security Triage Report question. The expected result should remain advisory-only and should not claim confirmed exploitation or real enforcement.

## Size Inspection Commands

Use these read-only commands before and after cleanup to confirm where disk space is going.

Directory size overview:

```powershell
Get-ChildItem -Force | Where-Object {$_.PSIsContainer} | ForEach-Object { $size=(Get-ChildItem $_.FullName -Recurse -Force -ErrorAction SilentlyContinue | Measure-Object Length -Sum).Sum; [PSCustomObject]@{Name=$_.Name; SizeGB=[math]::Round($size/1GB,2)} } | Sort-Object SizeGB -Descending
```

Largest files:

```powershell
Get-ChildItem -Recurse -File -Force -ErrorAction SilentlyContinue | Sort-Object Length -Descending | Select-Object -First 30 FullName,@{Name="SizeMB";Expression={[math]::Round($_.Length/1MB,2)}}
```

## Git Hygiene

Recommended read-only checks:

```powershell
git status --short --untracked-files=all
git status --ignored --short
git count-objects -vH
```

Guidance:

- Ignored caches should not be committed.
- If a cache appears as untracked rather than ignored, review whether `.gitignore` already covers it before changing ignore rules.
- Do not modify `.gitignore` as part of cleanup unless the missing ignore rule is clear and reviewed.
- `git count-objects -vH` helps identify loose object/pack size, but do not run `git gc` during a release/demo window unless explicitly planned.

## Recommended Cleanup Flow

1. Check git status:

   ```powershell
   git status --short --untracked-files=all
   ```

2. Inspect sizes with the read-only size commands above.
3. Remove safe caches only: `.mypy_cache`, `.pytest_cache`, `.ruff_cache`, `.pytest_tmp`, and recursive `__pycache__`.
4. Recheck sizes.
5. Run a focused smoke test, such as:

   ```powershell
   python -m pytest tests/test_ui_demo_scenarios.py
   ```

6. Keep `venv`, `chroma_db`, screenshots, and demo evidence unless cleanup was explicitly planned.
7. Recheck git status and confirm no generated cache files are staged or accidentally tracked.

## What Not To Do

- Do not run broad recursive delete commands from the wrong directory.
- Do not delete `.git`.
- Do not delete `knowledge`.
- Do not delete `docs/screenshots` before demo/report review.
- Do not delete `chroma_db` without planning to rebuild it.
- Do not delete `venv` right before a demo.
- Do not commit generated cache files.
- Do not treat disk cleanup as a fix for startup slowness; lazy-loading heavy RAG/embedding imports is the real performance direction from v2.8-A.
