# v1.5 Spec — ControllerAgent and Tool Registry

## Purpose

v1.5 builds typed agent infrastructure: Tool Registry, deterministic ControllerAgent dispatch, and wrappers around existing safe local project capabilities. It does not introduce LLM-driven routing or new analysis skills.

ControllerAgent is a router/orchestrator, not the final security judge. Final detection, risk, and decision remain deterministic and policy-controlled. LLM may help explain or assist through existing paths, but it must not select tools, override final verdicts, or create new analysis behavior.

This milestone prepares safe typed infrastructure before any future Smart Router or Auto Route UX work.

## Problem

The current CLI asks users to choose a mode manually. This is simple and acceptable for earlier prototype phases, but it creates friction as the project gains more capabilities.

Users may not know whether their input is:

- a payload or event input
- a raw log line
- a log file path
- a security knowledge question
- a report-aware follow-up question

Existing capabilities are also spread across mode handlers and agent functions. There is no typed central tool registry describing available tools, inputs, outputs, and safety boundaries.

v1.5 aims to make routing more structured while preserving the deterministic safety model.

## Goals

- Add a typed Tool Registry.
- Define tool contracts using Pydantic input/output models.
- Add ControllerAgent routing layer.
- Add deterministic mode-to-tool dispatch for registered skills.
- Wrap only existing capabilities as skills.
- Add no new analysis capability in v1.5.
- Preserve deterministic final decision boundaries.
- Keep tests deterministic and CI-friendly.
- Avoid requiring Ollama / RAG initialization for core routing tests.
- Keep full Smart Router and Auto Route UX deferred to v1.7.

## Non-Goals

- No autonomous offensive actions.
- No real firewall / WAF / EDR / SIEM / SOAR actions.
- No LLM-controlled final verdict.
- No arbitrary tool execution.
- No web browsing or internet lookup.
- No long-running autonomous investigation loop.
- No full multi-agent red/blue simulation in v1.5.
- No UI/dashboard yet.
- No Auto Route UX.
- No Investigation Planner.
- No Rule Explainer.
- No new AI skills.

## Proposed Architecture

```text
User Input / Existing CLI Mode
→ ControllerAgent
→ Deterministic Mode-to-Tool Dispatch
→ ToolRegistry
→ Selected Registered Skill
→ Existing Deterministic Pipeline / RAG / Report Follow-up
→ Unified Output
```

Full Smart Router is deferred to v1.7. v1.5 dispatch may use explicit mode / route hints rather than free-form input classification.

Role separation:

- Deterministic mode-to-tool dispatch maps explicit routes or existing CLI modes to registered skills.
- ToolRegistry stores available tools and contracts.
- ControllerAgent coordinates routing and result formatting.
- SecurityAgent / TriagePolicy still own security triage logic.
- LLMAssist remains advisory only.
- RAG remains explanation-only.

## Tool Registry

A Tool Registry is a typed catalog of safe local project capabilities. It should describe what each tool accepts, what it returns, and what safety boundaries apply before the controller is allowed to call it.

A tool entry should include:

- `name`
- `description`
- `input_model`
- `output_model`
- `handler`
- `safety_level`
- `requires_llm`
- `requires_rag`
- `deterministic`
- `allowed_modes` / `input_kinds`
- `notes`

Example tool entries:

- `payload_triage`
- `raw_log_translate`
- `log_file_ingest`
- `rag_security_qa`
- `report_followup`
- `incident_json_export`

v1.5 registered skills are exactly these six wrapper skills:

- `payload_triage`
- `raw_log_translate`
- `log_file_ingest`
- `rag_security_qa`
- `report_followup`
- `incident_json_export`

Explicitly deferred:

- `rule_explainer` -> v1.6 RAG v2
- `investigation_planner` -> v1.8 or later

v1.5 skill requirements:

- wraps an existing module/function
- does not introduce new detection, RAG, or LLM logic
- uses typed input/output contracts
- has unit tests
- does not call new LLM paths
- does not add new dependencies

Conceptual pseudo-structure only:

```python
ToolSpec(
    name="payload_triage",
    input_model=PayloadTriageInput,
    output_model=SecurityTriageOutput,
    handler=run_payload_triage,
    deterministic=True,
    requires_llm=False,
    requires_rag=False,
    safety_level="safe_local_analysis",
)
```

This is a design target for v1.5 implementation phases, not an implementation in this spec.

## Deterministic Dispatch in v1.5

v1.5 does not implement a full Smart Router. It may use explicit route hints or existing CLI mode mapping to choose a registered skill. Full free-form input classification and routing accuracy evaluation are deferred to v1.7. LLM-based routing is also deferred.

Deferred v1.7 Smart Router input kinds may include:

### `payload_or_event`

Examples:

```text
<script>alert(1)</script>
test; rm -rf /tmp/test
' OR '1'='1
```

### `raw_log_line`

Example:

```text
2026-05-01T10:00:00Z login_failed src_ip=10.0.0.5 user=admin endpoint=/login status=401
```

### `log_file_path`

Examples:

```text
demo_logs/scenario_a_mixed_auth.log
<path-to-auth.log>
```

### `security_knowledge_question`

Examples:

```text
brute force 是什麼？
SQL Injection 要怎麼防？
```

### `report_followup`

Examples:

```text
EV-003 是什麼意思？
為什麼是 MONITOR？
我接下來要查什麼？
```

### `unknown`

Future fallback route:

- ask the user to clarify, or
- ask the user to choose a mode.

Important:

- v1.5 dispatch should be deterministic and explicit.
- v1.5 should not infer arbitrary free-form intent.
- Full routing accuracy evaluation is deferred to v1.7.
- LLM-based routing remains deferred.

## Pydantic Boundary Models

Planned input models:

- `ControllerInput`
- `RouterDecision`
- `PayloadTriageInput`
- `RawLogInput`
- `LogFileInput`
- `KnowledgeQuestionInput`
- `ReportFollowupInput`

Planned output models:

- `ControllerOutput`
- `ToolExecutionResult`
- `RouteExplanation`

Suggested fields:

```python
class ControllerInput(BaseModel):
    raw_text: str
    context: dict[str, Any] = {}
    last_incident_id: str | None = None


class RouterDecision(BaseModel):
    input_kind: str
    selected_tool: str | None
    confidence: str
    reason: str
    requires_clarification: bool = False


class ControllerOutput(BaseModel):
    status: str
    selected_tool: str | None
    response_text: str
    structured_result: dict[str, Any]
    route: RouterDecision
    warnings: list[str]
```

These models should stay small at first. The purpose is to create a typed boundary, not to overdesign a full agent framework.

## Safety Rules

ControllerAgent must not:

- override TriagePolicy
- alter final Risk Level or Decision
- run arbitrary shell commands
- execute unregistered tools
- treat RAG output as detection
- treat LLMAssist output as final verdict
- perform real enforcement action

ControllerAgent may:

- choose a safe local analysis path
- explain why a route was selected
- ask for clarification
- return structured outputs
- call deterministic local tools registered in ToolRegistry

### Phase 5A-6 — Documentation Sync

## Test Strategy

Required tests:

- `ControllerInput` validation.
- `ToolSpec` validation.
- Tool registry can register and retrieve tools.
- Duplicate tool names rejected.
- Six wrapper skills have valid `ToolSpec`.
- Each wrapper validates input/output contracts.
- ControllerAgent dispatches by explicit route/mode.
- ControllerAgent rejects unregistered tools.
- ControllerAgent does not override deterministic decision.
- Existing 159 tests still pass.

Deferred to v1.7:

- router detects payload inputs.
- router detects raw auth log lines.
- router detects log file paths.
- router detects security questions.
- router detects `EV-ID` / `F-ID` follow-up.
- router returns clarification for unknown free-form input.

Quality gate:

```text
python -m pytest
python -m ruff check .
python -m mypy app.py modules tests
```

Tests must not require:

- Ollama
- Chroma
- embeddings
- Torch
- ChatOllama
- app startup

## Acceptance Criteria

v1.5 is complete when:

- Tool Registry exists with typed `ToolSpec` entries.
- Six wrapper skills are registered and tested.
- ControllerAgent can deterministically dispatch safe local requests by explicit route/mode.

- Final verdict remains deterministic and policy-controlled.
- Full test suite passes.
- Documentation is updated.
