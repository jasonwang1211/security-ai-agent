# v1.4 Spec — Detection-as-Code Lite

## Purpose

Move rule-based attack signatures out of Python hard-coded lists into YAML-based detection rule files, while preserving deterministic detection behavior.

This milestone should improve:

- maintainability
- rule metadata
- testing discipline
- future evaluation support
- portfolio-grade detection engineering story

This is not ML or AI detection. The detection layer remains rule-based and deterministic.

## Problem

The current detector is acceptable for earlier prototype phases, but it has several limitations as the project grows:

- Existing attack signatures are hard-coded in Python.
- Adding a new detection requires modifying production code.
- Rule metadata is limited or scattered.
- It is harder to attach severity, confidence, references, and MITRE techniques to each rule.
- This makes future benchmark and per-rule evaluation harder.

v1.4 will make detection rules more maintainable without weakening the existing deterministic safety boundary.

## Goals

- Add YAML-based rule files.
- Add a rule loader with schema validation.
- Preserve existing detector behavior.
- Migrate existing XSS / SQL Injection / Path Traversal / Command Injection signatures.
- Add rule metadata:
  - `id`
  - `title`
  - `attack_type`
  - `severity`
  - `confidence`
  - `patterns`
  - `match_mode`
  - `case_sensitive`
  - `mitre_techniques`
  - `references`
  - `notes`
- Keep tests deterministic and CI-friendly.

## Non-Goals

- No ML anomaly detection.
- No real WAF / firewall enforcement.
- No hot reload.
- No cloud rule sync.
- No Sigma / YARA full compatibility in v1.4.
- No broad log detection language yet.
- No LLM-generated rules.
- No automatic rule deployment.

## YAML Rule Schema

Proposed schema:

```yaml
id: CMD-001
title: Basic Command Injection
attack_type: Command Injection
severity: HIGH
confidence: 0.9
patterns:
  - "; rm "
  - "; rm -rf"
  - "&&"
  - "||"
  - "| nc "
  - "$("
  - "`"
match_mode: substring
case_sensitive: false
mitre_techniques:
  - T1059
references:
  - knowledge/blue_team/attack_techniques/command_injection.md
notes: "Basic command injection indicators used by the deterministic detector."
enabled: true
```

### Field Rules

| Field | Rule |
|---|---|
| `id` | Required stable rule ID. Must be unique across loaded rules. |
| `title` | Required human-readable rule title. |
| `attack_type` | Required report-facing attack type. |
| `severity` | Required. Must be one of `LOW`, `MEDIUM`, or `HIGH`. |
| `confidence` | Required float between `0.0` and `1.0`. |
| `patterns` | Required non-empty list of strings. |
| `match_mode` | Required. Initially `substring` or `regex`. |
| `case_sensitive` | Optional bool. Defaults to `false`. |
| `mitre_techniques` | Optional list. |
| `references` | Optional list. |
| `notes` | Optional string. |
| `enabled` | Optional bool. Defaults to `true`. |

v1.4 should start with `substring` mode first. `regex` can exist in the schema, but implementation may be limited or deferred if it increases complexity.

## Rule Directory Structure

Recommended:

```text
detections/
└── blue_team/
    ├── xss/
    │   └── xss_basic.yml
    ├── sql_injection/
    │   └── sql_injection_basic.yml
    ├── path_traversal/
    │   └── path_traversal_basic.yml
    └── command_injection/
        └── command_injection_basic.yml
```

`detections/blue_team/` keeps these as defensive detection rules. Folder names should be stable and lowercase. File names should be descriptive. Future versions may add rules by technique, source, or environment.

## Architecture

Proposed flow:

```text
detections/*.yml
-> DetectionRule model
-> RuleLoader
-> RuleBasedDetector
-> DetectorResultModel
-> TriagePolicy
```

Suggested modules:

- `modules/detection_rules.py`
  - Pydantic `DetectionRule`
  - validation
- `modules/rule_loader.py`
  - load YAML rule files
  - skip disabled rules
  - fail safely on invalid rules
- `modules/detector.py`
  - keep public behavior stable
  - use loaded rules instead of hard-coded signatures

The public detector output should remain backward-compatible:

- `status`
- `attack_types`
- `matched_signatures`
- `original_input`

New metadata can be added later, but existing tests should not break.

## Behavior Compatibility

v1.4 should preserve current detector behavior for existing demo and regression cases:

- XSS remains detected through existing XSS indicators.
- SQL Injection remains detected through existing SQL Injection indicators.
- Path Traversal remains detected through existing Path Traversal indicators.
- Command Injection remains detected through existing Command Injection indicators.
- Current risk and decision behavior should remain deterministic and test-covered.

Any intentional behavior change should be explicitly documented and covered by tests.

## Migration Plan

### Phase 4A-1 — Rule Schema and Loader

- Add `DetectionRule` model.
- Add YAML loading utility.
- Add validation tests.
- Add one sample rule file.
- Do not change detector runtime yet.

### Phase 4A-2 — Detector Adapter

- Update detector to use loaded YAML rules.
- Preserve existing behavior.
- Existing golden tests must still pass.

### Phase 4A-3 — Rule Migration

- Move XSS rules to YAML.
- Move SQL Injection rules to YAML.
- Move Path Traversal rules to YAML.
- Move Command Injection rules to YAML.
- Remove duplicated hard-coded lists only after tests prove parity.

### Phase 4A-4 — Metadata and Reporting

- Add optional rule metadata to detector output if safe.
- Include severity / confidence / MITRE techniques in internal result or metadata.
- Do not break current Security Triage Report.

### Phase 4A-5 — Documentation and Demo Sync

- Update README / REPORT / demo_outputs / ROADMAP.
- Add Detection-as-Code demo section.
- Tag v1.4.0 after release gate.

## Test Strategy

Required tests:

- Valid YAML rule loads successfully.
- Invalid YAML rule fails validation.
- Disabled rule is not used.
- Missing required fields fail validation.
- Confidence outside `0.0`-`1.0` fails validation.
- `substring` match works.
- Case-insensitive matching works.
- `regex` matching either works or is explicitly deferred.
- Existing golden payload tests still pass.
- Command Injection regression remains passing.
- Benign input still does not alert.

Quality gate:

```powershell
python -m pytest
python -m ruff check .
python -m mypy app.py modules tests
```

Tests must not initialize:

- RAGQA
- Chroma
- embeddings
- Torch
- Ollama
- ChatOllama

## Acceptance Criteria

v1.4 is complete when:

- Existing hard-coded XSS / SQL Injection / Path Traversal / Command Injection signatures are represented in YAML.
- Detector behavior remains compatible with current tests.
- Rule loader validates YAML rules.
- Disabled rules are ignored.
- Invalid rule files fail safely.
- Golden tests still pass.
- Scenario A v1.3 integration test still passes.
- Documentation is updated.
- v1.4.0 tag can be created.

## Deferred Items

Deferred beyond v1.4:

- Full Sigma compatibility.
- YARA-like rule engine.
- Hot reload.
- UI rule editor.
- Rule marketplace / remote sync.
- LLM-generated detection rules.
- ML anomaly detection.
- Production SIEM integration.
- Automatic enforcement.

## Open Questions

- Should invalid YAML fail the whole loader or be skipped with warnings?
- Should detector load rules at import time, constructor time, or first use?
- Should rule metadata be exposed in `DetectorResultModel` in v1.4?
- Should `regex` `match_mode` be implemented immediately or deferred?
- Should rules live under `detections/blue_team/` or `knowledge/blue_team/detection_rules/`?
- Should rule references point to local KB docs, external URLs, or both?
