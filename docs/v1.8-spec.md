# v1.8 Spec — Protected Runtime Wiring and Analyst UX

## 1. Purpose

Define v1.8 as the first controlled user-facing integration milestone after v1.7.

v1.8 focuses on:

- protected runtime wiring
- source-cited report/rule explanation path
- AnswerGuardrails applied to helper outputs
- Smart Router preview mode
- analyst UX improvements
- no broad runtime replacement

Important framing:

- v1.8 must not replace `RAGQA` wholesale.
- v1.8 must not make Smart Router the default CLI path immediately.
- v1.8 must not introduce LLM routing.
- v1.8 must not let AI override Risk Level or Decision.
- v1.8 keeps deterministic detector and policy output as the final security authority.
- v1.8 keeps LLM/RAG advisory and explanation-only.
- v1.8 keeps BLOCK, MONITOR, and ALLOW simulated.

## 2. Problem

v1.7 added the safety, evaluation, and router foundations needed before exposing more helper output to analysts. The repository now has deterministic AnswerGuardrails, eval cases, an eval runner, and an isolated rule-based Smart Router foundation.

Smart Router exists, but it is not CLI-wired. AnswerGuardrails exist, but they are not yet applied to user-facing helper outputs. RAG v2 report and rule explainers exist, but they are staged rather than part of the primary runtime.

Users still need clearer source-cited report/rule explanations and safe next-step guidance after viewing reports. The next step should be narrow protected wiring through existing ownership boundaries, not another layer of helper modules.

## 3. Goals

1. Add protected report/rule explanation path.
2. Apply AnswerGuardrails to source-cited helper outputs.
3. Add Smart Router preview mode without replacing CLI modes.
4. Add analyst follow-up suggestions after reports.
5. Preserve existing CLI modes.
6. Preserve deterministic final verdict authority.
7. Keep LLM/RAG advisory only.
8. Keep BLOCK, MONITOR, and ALLOW simulated.

## 4. Non-Goals

v1.8 does not include:

- No full RAGQA replacement.
- No Smart Router default auto-route mode.
- No LLM-based routing.
- No LLM final verdict.
- No real firewall, WAF, SIEM, SOAR, or cloud enforcement.
- No autonomous Investigation Planner yet.
- No automatic rule generation.
- No external threat intelligence.
- No dashboard.
- No large module consolidation unless required by protected wiring.
- No new parallel RAG modules without clear ownership.

## 5. Proposed Phase Plan

### Phase 8A-1 — Protected Report/Rule Explanation Path

- Use existing `rag_explainers.py`.
- Use existing `report_followup.py` as the likely integration point.
- Do not replace general `RAGQA`.
- Add tests for EV-ID / F-ID / rule explanation.

### Phase 8A-2 — Guarded Answer Output

- Apply `answer_guardrails.py` to source-cited helper outputs.
- Unsafe helper output should return a safe fallback.
- No LLM safety classifier.

### Phase 8A-3 — Smart Router Preview Mode

- Add a non-executing preview path.
- Show selected route details.
- Do not execute tools automatically.
- Preserve existing CLI modes.

### Phase 8A-4 — Analyst UX Suggestions

- Show deterministic follow-up questions after report output.
- Base suggestions on report sections, IDs, and decision type.
- Do not use LLM generation.

### Phase 8A-5 — Documentation and Release Gate

- Update README / REPORT / ROADMAP / TECH_NOTES.
- Run the full quality gate.
- Prepare the v1.8.0 tag.

## 6. Protected Runtime Wiring Strategy

Keep `RAGQA` active for general knowledge Q&A. v1.8 should not replace the existing general RAG path or make staged RAG v2 helpers the primary runtime for every question.

Wire RAG v2 helpers only into narrow report/rule explanation flows. Prefer `report_followup.py` as the first user-facing integration point because it already owns report-aware EV/F-ID follow-up. Use existing `rag_explainers.py` for source-cited report/rule explanations rather than creating a parallel RAG module.

Apply AnswerGuardrails before returning helper output to the user. Guardrails check the answer text and source-cited helper output; they do not alter Risk Level, Decision, evidence, finding severity, or detector results. If guardrails fail, return a safe fallback that states the limitation and avoids unsupported enforcement or verdict claims.

The protected helper path must stay light. Do not call Chroma, Ollama, embeddings, Torch, ChatOllama, or an LLM from this path. Source-cited helper output should be deterministic and testable.

## 7. Smart Router Scope

Smart Router in v1.8 may be exposed as preview/debug mode only.

The preview path should show:

- `input_kind`
- `route`
- `confidence`
- `reason`

The preview path must not execute the selected route automatically. It must not replace numbered CLI modes, change Risk Level or Decision, or use LLM routing. Route selection is advisory UI guidance; it is not a security verdict and not an enforcement action.

## 8. Analyst UX Scope

Add deterministic follow-up suggestions after report output. Suggestions should help users ask better questions, not perform actions.

Suggested questions may include:

- 為什麼是 MONITOR？
- EV-003 是什麼意思？
- 下一步要查什麼？
- 這條規則為什麼命中？

Suggestions should be selected from report structure, EV/F/rule IDs if present, and decision type. They should not be generated by an LLM, should not trigger tools automatically, and should not imply real remediation or enforcement.

## 9. Safety Boundaries

Preserve these boundaries:

- deterministic detector and policy own final verdict
- LLM/RAG are advisory/explanation-only
- RAG is not a detection source
- Smart Router selects route only
- AnswerGuardrails check output, not verdict
- BLOCK, MONITOR, and ALLOW are simulated
- no real firewall, WAF, SIEM, SOAR, cloud, or endpoint action is performed
- no offensive automation
- no automatic detection rule activation

## 10. Test Strategy

v1.8 tests should cover:

- protected report/rule explanation outputs
- AnswerGuardrails pass/fail behavior
- safe fallback when output is unsafe
- Smart Router preview does not execute tools
- existing CLI modes remain available
- no RAGQA replacement
- no Chroma/Ollama/Torch startup in unit tests
- suggested follow-up questions are deterministic
- existing 445 tests still pass

## 11. Acceptance Criteria

v1.8 is complete when:

- Protected report/rule explanation path exists.
- AnswerGuardrails are applied to helper outputs.
- Unsafe helper output has safe fallback.
- Smart Router preview mode exists or is documented as deferred.
- Existing CLI modes still work.
- General RAGQA remains active and unchanged unless explicitly scoped.
- Full quality gate passes.
- Documentation is updated.
- v1.8.0 tag can be created.

## 12. Deferred Items

Deferred beyond v1.8:

- full Smart Router auto-route CLI
- full RAGQA replacement
- autonomous Investigation Planner
- LLM-based router
- dashboard
- external threat intelligence
- large statistical benchmark
- production SIEM/SOAR integration
- real enforcement

## 13. Open Questions

- Should protected report/rule explanation live in `report_followup.py` or a small wrapper?
- Should Smart Router preview be a new CLI option or developer-only helper?
- Should AnswerGuardrails block output or downgrade confidence?
- Should unsafe output return fallback text or structured error?
- Should follow-up suggestions be generated from report object or rendered text?
- Should v1.8 include any lazy initialization work, or defer it?
