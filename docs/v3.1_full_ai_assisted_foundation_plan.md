# v3.1 Full AI-Assisted Foundation Plan

## Scope

v3.1 adds backend contracts for a full AI-assisted advisory path. It does not redesign the Streamlit layout, change screenshots, alter detector behavior, modify retrieval semantics, rewrite graph construction, or add enforcement behavior.

The implementation is intentionally contract-first:

- prompt safety contract for defensive SOC analyst assistance;
- provider abstraction for disabled, local, and OpenAI-compatible modes;
- deterministic fallback for every provider failure path;
- structured full AI-assisted result contract;
- event-aware Q&A backend over an existing EvidenceGroundingBundle.

## Safety Boundary

The deterministic security path remains the authority:

- Rule-Based Detector remains the detection authority.
- Risk Level / Decision remain deterministic and must be copied exactly.
- BLOCK / MONITOR / ALLOW remain simulated decisions only.
- LLM, RAG, AI Brief, Evidence Gap, Similar Cases, and Graph context are advisory only.
- Similar Cases cannot prove compromise or successful execution.
- Graph context is not a detection source.
- No real firewall, WAF, EDR, account, cloud, SIEM, or SOAR action is executed.
- No exploit, PoC, traffic generation, load testing, or offensive automation is provided.
- Human review is required.

## Provider Model

The provider layer is disabled by default through `SENTINEL_AI_PROVIDER_MODE`.

Supported modes:

- `disabled`: default; returns deterministic fallback without provider execution.
- `fake`: injection-only provider for tests; not enabled from normal environment runtime.
- `local`: optional local provider; lazy import happens only inside the provider call path.
- `openai_compatible`: optional HTTP chat-completions compatible provider using the Python standard library.

CI and automated tests must not require a live LLM, API key, Ollama, Chroma, embeddings, or network access. Missing provider configuration is represented as provider failure data and must not raise to callers.

## Full AI-Assisted Result Contract

The full AI-assisted result copies the official verdict from the EvidenceGroundingBundle and never allows provider output to overwrite it. Provider output must pass JSON/schema validation, citation validation, and advisory guardrails before it is accepted.

Fallback is required for:

- provider disabled or unavailable;
- invalid JSON;
- missing or unknown citations;
- unsafe wording;
- verdict tampering;
- treating Similar Cases as proof;
- treating Graph context as a detection source;
- real enforcement wording;
- exploit, PoC, traffic generation, load testing, or offensive automation wording.

Fallback output remains deterministic, advisory, and review-gated.

## Event-Aware Q&A Backend

The event-aware Q&A backend answers questions about a current EvidenceGroundingBundle. It may use deterministic verdict data, rule IDs, evidence IDs, evidence gaps, already-provided RAG context, already-provided similar-case context, and already-provided graph context.

It does not add a controller skill, UI wiring, live retrieval, graph traversal, or write behavior in this milestone.

Unsafe questions are refused before provider execution. Provider answers are accepted only when they preserve known citation IDs and pass safety guardrails.

## Deferred Work

The following work is intentionally deferred:

- Streamlit provider selector;
- UI redesign;
- screenshot refresh;
- report export polish;
- controller skill wiring;
- live provider smoke tests;
- richer analyst conversation state.

## Validation Expectations

Release validation for this foundation should include:

- full pytest suite;
- ruff check;
- mypy over `app.py`, `modules`, and `tests`;
- `git diff --check`;
- wording scan for disallowed vendor or trailer text.

A passing gate means the backend contracts are ready for review on the v3.1 branch. It does not mean the optional provider path has been live-tested against a production model.
