# v1.3 Spec — Evidence and Incident Capability

## Goal

Move the project from single-event triage toward incident-style evidence handling while preserving deterministic safety boundaries.

## Non-Goals

- No real firewall / WAF / EDR / SIEM / SOAR action.
- No offensive automation.
- No LLM override of final triage decisions.
- No production-grade SIEM replacement.

## Target Scenario

Possible Account Compromise:

```text
Repeated login_failed events
-> same source_ip / user / target within a time window
-> followed by login_success
-> escalate to Possible Account Compromise
-> emit Security Triage Report and JSON Incident Report
```

## Report-aware RAG Follow-up

Core principle:
Report-aware RAG explains the existing report. It must not re-judge the verdict.

Responsibilities:

- Explain why the current report reached HIGH / MEDIUM / LOW.
- Explain why Decision is MONITOR / BLOCK / ALLOW.
- Explain evidence IDs such as EV-001.
- Explain finding IDs such as F-001.
- Explain AI Assist disagreement with final decision.
- Suggest next investigation steps based on current finding type.
- Cite KB docs or doc IDs.

Non-responsibilities:

- It must not override TriagePolicy.
- It must not claim the final verdict is wrong.
- It must not invent evidence.
- It must not make real enforcement recommendations.
- It must not replace analyst judgment.

WHAT / alternative / WHY distinction:

- TriagePolicy decides WHAT the final verdict is.
- LLMAssist suggests alternative interpretations during triage.
- Report-aware RAG explains WHY the verdict was reached after triage.

## LLM Safety Architecture

Mode:
v1.3 uses Advisory Scoring only.

LLM may output:

- `is_suspicious`
- `possible_attack_types`
- `confidence`
- `reasoning`
- `recommended_risk`
- `recommended_action`
- `evidence_references`

LLM may not:

- change final risk
- change final decision
- downgrade deterministic suspicious findings
- unilaterally decide BLOCK
- cite non-existent evidence IDs
- invent facts not present in EvidenceBundle or KB

Guardrails:

- evidence grounding
- no downgrade
- no unilateral BLOCK
- whitelist validation
- confidence sanity check
- prompt injection sanitizer
- audit log
- disagreement detector

Fallback:
If LLM output is invalid, unsafe, or ungrounded, fall back to deterministic results.

## Planned Capabilities

### 1. Time-window aggregation

Add time-window awareness to repeated authentication failures.

Initial idea:

- group by source_ip + target + optionally user
- threshold-based failed_count
- time window such as 5 minutes
- preserve related_events for evidence

### 2. Sequence-based finding

Detect suspicious sequence:

- many auth_failure events
- then login_success
- same user or same source_ip / target

Candidate output:

- `event_type`: `possible_account_compromise`
- `status`: `SUSPICIOUS` or `ALERT`
- `risk_level`: `HIGH`
- `decision`: `MONITOR` or `BLOCK` depending on policy

### 3. JSON Incident Report export

Add a structured export format using Pydantic boundary models.

Candidate fields:

- incident_id
- generated_at
- title
- status
- risk_level
- decision
- attack_type
- evidence
- timeline
- recommended_response
- simulation_notice

### 4. MITRE ATT&CK mapping

Initial mapping candidates:

- Brute Force -> T1110
- Valid Accounts -> T1078
- Account Discovery or Credential Access mappings can be considered later

Keep this as static mapping first; do not over-engineer.

## Testing Plan

Add deterministic tests before or with implementation:

- auth failures below threshold should not create incident
- auth failures over threshold should create brute_force_candidate
- failures followed by success should create possible_account_compromise
- JSON incident report validates against Pydantic model
- benign login_success alone should not create incident

Do not initialize RAG, Chroma, embeddings, Ollama, or real LLM clients in these tests.

## Deferred Items

v1.3 intentionally does not include:

- Smart Input Router
- Detection-as-Code YAML rule engine
- full multi-turn memory
- autonomous InvestigationAgent
- production SIEM/SOAR integration
- real enforcement actions
- broad real-world log format support
- Knowledge Graph
- fine-tuned embeddings
- Red / Blue simulation lab

These are intentionally deferred to avoid scope creep. v1.3 focuses on evidence, incident, LLM safety, JSON export, and report-aware follow-up.
