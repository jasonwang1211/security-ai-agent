# v1.5 Spec — ControllerAgent and Tool Registry

## Purpose

Move from a menu-first CLI workflow toward a typed controller layer that can route user input to the correct deterministic tool or analysis pipeline.

ControllerAgent is a router/orchestrator, not the final security judge. Final detection, risk, and decision remain deterministic and policy-controlled. LLM may help explain or assist, but it must not override final verdicts.

This milestone should reduce manual mode selection without creating unsafe agent behavior.

## Problem

The current CLI asks users to choose a mode manually. This is simple and acceptable for earlier prototype phases, but it creates friction as the project gains more capabilities.

Users may not know whether their input is:

- a payload or event input
- a raw log line
- a log file path
- a security knowledge question
- a report-aware follow-up question

Existing capabilities are also spread across mode handlers and agent functions. There is no typed central tool registry describing available tools, inputs, outputs, and safety boundaries.

v1.5 aims to make routing more structured while preserving the deterministic safety model.

## Goals

- Add a typed Tool Registry.
- Define tool contracts using Pydantic input/output models.
- Add ControllerAgent routing layer.
- Add a minimal deterministic routing layer for registered local tools.
- Defer full Smart Input Router UX polish and evaluation dataset to v1.6.
- Preserve deterministic final decision boundaries.
- Keep tests deterministic and CI-friendly.
- Avoid requiring Ollama / RAG initialization for core routing tests.

## Non-Goals

- No autonomous offensive actions.
- No real firewall / WAF / EDR / SIEM / SOAR actions.
- No LLM-controlled final verdict.
- No arbitrary tool execution.
- No web browsing or internet lookup.
- No long-running autonomous investigation loop.
- No full multi-agent red/blue simulation in v1.5.
- No UI/dashboard yet.

## Proposed Architecture

```text
User Input
→ ControllerAgent
→ MinimalRouter / SmartInputRouter-lite
→ ToolRegistry
→ Selected Tool
→ Deterministic Pipeline / RAG / Report Follow-up
→ Unified Output
```

Full Smart Input Router with classification evaluation is deferred to v1.6.

Role separation:

- MinimalRouter / SmartInputRouter-lite classifies the smallest useful input set for v1.5.
- ToolRegistry stores available tools and contracts.
- ControllerAgent coordinates routing and result formatting.
- SecurityAgent / TriagePolicy still own security triage logic.
- LLMAssist remains advisory only.
- RAG remains explanation-only.

## Tool Registry

A Tool Registry is a typed catalog of safe local project capabilities. It should describe what each tool accepts, what it returns, and what safety boundaries apply before the controller is allowed to call it.

A tool entry should include:

- `name`
- `description`
- `input_model`
- `output_model`
- `handler`
- `safety_level`
- `requires_llm`
- `requires_rag`
- `deterministic`
- `allowed_modes` / `input_kinds`
- `notes`

Example tool entries:

- `payload_triage`
- `raw_log_translate`
- `log_file_ingest`
- `rag_security_qa`
- `report_followup`
- `incident_json_export`

Conceptual pseudo-structure only:

```python
ToolSpec(
    name="payload_triage",
    input_model=PayloadTriageInput,
    output_model=SecurityTriageOutput,
    handler=run_payload_triage,
    deterministic=True,
    requires_llm=False,
    requires_rag=False,
    safety_level="safe_local_analysis",
)
```

This is a design target for v1.5 implementation phases, not an implementation in this spec.

## Minimal Router in v1.5

Initial route categories:

### `payload_or_event`

Examples:

```text
<script>alert(1)</script>
test; rm -rf /tmp/test
' OR '1'='1
```

### `raw_log_line`

Example:

```text
2026-05-01T10:00:00Z login_failed src_ip=10.0.0.5 user=admin endpoint=/login status=401
```

### `log_file_path`

Examples:

```text
demo_logs/scenario_a_mixed_auth.log
C:\Users\...\auth.log
```

### `security_knowledge_question`

Examples:

```text
brute force 是什麼？
SQL Injection 要怎麼防？
```

### `report_followup`

Examples:

```text
EV-003 是什麼意思？
為什麼是 MONITOR？
我接下來要查什麼？
```

### `unknown`

Fallback route:

- ask the user to clarify, or
- ask the user to choose a mode.

Important:

- v1.5 should implement only the smallest deterministic subset required for ControllerAgent integration.
- Full routing accuracy evaluation is deferred to v1.6.
- LLM-based routing remains deferred.

## Pydantic Boundary Models

Planned input models:

- `ControllerInput`
- `RouterDecision`
- `PayloadTriageInput`
- `RawLogInput`
- `LogFileInput`
- `KnowledgeQuestionInput`
- `ReportFollowupInput`

Planned output models:

- `ControllerOutput`
- `ToolExecutionResult`
- `RouteExplanation`

Suggested fields:

```python
class ControllerInput(BaseModel):
    raw_text: str
    context: dict[str, Any] = {}
    last_incident_id: str | None = None


class RouterDecision(BaseModel):
    input_kind: str
    selected_tool: str | None
    confidence: str
    reason: str
    requires_clarification: bool = False


class ControllerOutput(BaseModel):
    status: str
    selected_tool: str | None
    response_text: str
    structured_result: dict[str, Any]
    route: RouterDecision
    warnings: list[str]
```

These models should stay small at first. The purpose is to create a typed boundary, not to overdesign a full agent framework.

## Safety Rules

ControllerAgent must not:

- override TriagePolicy
- alter final Risk Level or Decision
- run arbitrary shell commands
- execute unregistered tools
- treat RAG output as detection
- treat LLMAssist output as final verdict
- perform real enforcement action

ControllerAgent may:

- choose a safe local analysis path
- explain why a route was selected
- ask for clarification
- return structured outputs
- call deterministic local tools registered in ToolRegistry

## Migration Plan

### Phase 5A-1 — Controller Types and ToolSpec

Add Pydantic controller input/output models. Add `ToolSpec` model. Add unit tests. No runtime routing yet.

### Phase 5A-2 — Tool Registry Skeleton

Add `modules/tool_registry.py`. Register existing local tools conceptually. Add tests for lookup, duplicate names, and required fields. No app wiring yet.

### Phase 5A-3 — Minimal Deterministic Router

Add deterministic router. Classify payload, raw log, file path, knowledge question, and report follow-up. Add tests. No LLM routing.

### Phase 5A-4 — ControllerAgent Skeleton

Add `modules/controller_agent.py`. Route to tool handlers through registry. Use dummy handlers or thin wrappers first. Preserve existing CLI modes.

### Phase 5A-5 — CLI Optional Auto Mode

Optional Auto Route prototype may be added only if 5A-1 through 5A-4 remain stable; otherwise defer CLI auto mode to v1.6.

Possible optional menu item:

```text
5. Auto route input
```

Do not remove existing modes. Add demo and tests.

### Phase 5A-6 — Documentation and Release Gate

Update README / REPORT / demo_outputs / ROADMAP / ARCHITECTURE_DEBT. Add demo case. Run full quality gate. Tag `v1.5.0`.

## Test Strategy

Required tests:

- `ControllerInput` validation.
- `ToolSpec` validation.
- Tool registry can register and retrieve tools.
- Duplicate tool names rejected.
- Router detects payload inputs.
- Router detects raw auth log lines.
- Router detects log file paths.
- Router detects security questions.
- Router detects `EV-ID` / `F-ID` follow-up.
- Router returns clarification for unknown input.
- ControllerAgent does not override deterministic decision.
- Existing 141 tests still pass.

Quality gate:

```text
python -m pytest
python -m ruff check .
python -m mypy app.py modules tests
```

Tests must not require:

- Ollama
- Chroma
- embeddings
- Torch
- ChatOllama
- app startup

## Acceptance Criteria

v1.5 is complete when:

- Tool Registry exists with typed tool specs.
- Minimal deterministic router can classify the initial safe local input kinds needed for registered tools.
- ControllerAgent can route safe local requests.
- Existing CLI modes remain available.
- Optional Auto Route mode exists or is clearly deferred.
- Final verdict remains deterministic.
- Full Smart Input Router evaluation is deferred to v1.6.
- Full test suite passes.
- Documentation is updated.
- `v1.5.0` tag can be created.

## Deferred Items

Deferred beyond v1.5:

- LLM-based routing.
- Full Smart Input Router UX and evaluation dataset.
- Routing accuracy benchmark.
- Multi-step autonomous investigation.
- Full web dashboard.
- Real SIEM / SOAR integration.
- External API enrichment.
- Red/Blue simulation lab.
- Long-term memory.
- Automated response execution.
- Production deployment.

## Open Questions

- Should report-followup routing require a stored last Incident?
- How should the controller handle multiple possible routes?
- Should file path detection check filesystem existence immediately?
- Should RAG/LLM tools be lazy-loaded through ToolRegistry?
